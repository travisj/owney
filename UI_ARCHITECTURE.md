# Owney Test UI - Architecture

## Overview

A lightweight React + TypeScript test interface for validating Owney mailserver features without a full UI implementation. Built with Vite for fast development and production builds.

## Stack

- **Frontend**: React 18 + TypeScript
- **Bundler**: Vite 5 (very fast builds)
- **Styling**: Plain CSS (no framework, minimal dependencies)
- **Storage**: Browser localStorage (tokens)
- **API**: Direct JMAP calls via fetch

## Design Philosophy

1. **Minimal Dependencies**: Only React + Vite + TypeScript
2. **Testing First**: Quick way to test new features before UI exists
3. **Foundation**: Designed to be replaced by a real UI later
4. **No Styling Framework**: CSS is hand-written for simplicity

## Build Process

1. **Development**: `npm run dev` → Vite dev server at localhost:5173
2. **Production**: `npm run build` → Outputs to `crates/owney-api/static/`
3. **Runtime**: Rust server serves static files from that directory

### Build Output

Vite generates:
- Single `index.html` with correct asset references
- Hashed asset files (prevents caching issues)
- Minified JS/CSS bundles
- Automatically routes `/` to `index.html`

## How the API Works

### Authentication Flow

1. User gets JMAP bearer token from server (admin/automation)
2. Pastes token into UI
3. Stored in `localStorage` (persists across reloads)
4. All subsequent API calls include: `Authorization: Bearer <token>`

### JMAP Client (`api.ts`)

```typescript
// Simple wrapper around fetch + JMAP protocol
callJmap(methodCalls: Array<[string, args, id]>) 
  → sends to /jmap/api
  → waits for response
  → returns raw JMAP response

getSession() 
  → fetches from /.well-known/jmap
  → returns account info + capabilities
```

The client is intentionally simple to keep dependencies minimal.

### Feature Testers (`EmailTester.tsx`, `CalendarTester.tsx`)

Each tester:
1. Has a button for each operation
2. Calls `getSession()` to find account ID
3. Calls `callJmap()` with method calls
4. Displays raw JSON response
5. Shows success/error status

This pattern makes it easy to add new testers:

```tsx
const handleGetSomething = async () => {
  try {
    const session = await getSession()
    const accountId = Object.keys(session.accounts)[0]
    const result = await callJmap([
      ['Something/get', { accountId }, '0'],
    ])
    setStatus('✓ Retrieved')
    setResponse(JSON.stringify(result, null, 2))
  } catch (error) {
    setStatus(`✗ ${error.message}`)
  }
}
```

## Integration with Rust Backend

### Static File Serving

In `crates/owney-api/src/lib.rs`:

```rust
use tower_http::services::ServeDir;

pub fn router(state: Arc<ApiState>) -> Router {
    let static_dir = std::env::var("UI_STATIC_DIR")
        .unwrap_or_else(|_| "./static".to_string());
    
    Router::new()
        .route("/healthz", get(healthz))
        .route("/.well-known/jmap", get(session))
        // ... other routes ...
        .fallback(ServeDir::new(&static_dir).append_index_html_on_directories(true))
        .with_state(state)
}
```

**Key Points:**
- `fallback()` catches all routes not handled by explicit handlers
- `append_index_html_on_directories(true)` enables SPA routing
- All `/jmap/*` calls hit the API handler (higher priority)
- All other requests try `static/` (fallback)
- If file not found in static/, returns 404

### Environment Variable

```bash
UI_STATIC_DIR=./static cargo run  # explicit path (default)
UI_STATIC_DIR=/usr/share/owney-ui/static cargo run  # custom path
```

## Styling

### Global Styles (`index.css`)

- CSS variables for colors (future: easy dark mode)

## Adding New Features

### To Add an Email Tester Button

1. Add to `EmailTester.tsx`:

```tsx
const handleCustomOperation = async () => {
  setLoading(true)
  setStatus('')
  try {
    const session = await getSession()
    const accountId = Object.keys(session.accounts)[0]
    const result = await callJmap([
      ['Email/custom', { accountId, ... }, '0'],
    ])
    setStatus('✓ Operation completed')
    setResponse(JSON.stringify(result, null, 2))
  } catch (error) {
    setStatus(`✗ ${error.message}`)
  } finally {
    setLoading(false)
  }
}

// In JSX:
<button onClick={handleCustomOperation} disabled={!token || loading}>
  {loading ? 'Loading...' : 'Custom Operation'}
</button>
```

### To Add a New Feature Tab

1. Create `ui/src/components/NewFeatureTester.tsx` (follow EmailTester pattern)
2. Import in `App.tsx`
3. Add tab button for the new feature
4. Add conditional render in tab-content

## Development Tips

### Hot Reload

```bash
cd ui && npm run dev
# Visit localhost:5173
# Changes to src/ reload instantly
```

### Check for TypeScript Errors

```bash
cd ui && npx tsc --noEmit
```

### Debug JMAP Responses

- Open browser DevTools (F12)
- Go to Network tab
- Filter by "jmap"
- Click requests to see full request/response
- Or: Check the UI's response box (already displayed)

### Testing Locally

1. Create test account on server
2. Get bearer token
3. Paste into UI
4. Use UI to trigger operations
5. Check server logs: `RUST_LOG=owney=debug cargo run`

## Performance Notes

- **Bundle Size**: ~50KB gzipped (React + app code)
- **Load Time**: <1s on modern connection
- **Runtime**: Minimal (no heavy libraries)
- **Rendering**: Instant (no animations)

## Security Notes

- **Tokens in localStorage**: OK for test UI (don't use for production)
- **No HTTPS enforcement**: Add in reverse proxy
- **No rate limiting**: Server should handle rate limiting
- **CORS**: Not needed (same origin or via reverse proxy)

## Troubleshooting

| Issue | Solution |
|-------|----------|
| `Cannot find static/` | Run `npm run build` in ui/ |
| TypeScript errors | Run `npx tsc --noEmit` in ui/ |
| Port already in use | `LISTEN_ADDR=127.0.0.1:9999 cargo run` |
| Token not working | Check token is valid: `curl -H "Authorization: Bearer TOKEN" http://localhost:8008/.well-known/jmap` |
| UI not loading | Check browser console (F12) for JS errors |
| 404 on /jmap/api | Make sure Rust server is running |
