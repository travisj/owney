## What's enforced on `main` going forward

- Workspace-wide `cargo clippy --all-targets -- -D warnings` is green
  and stays so after every merge.
- Every crate has a typed id surface (`EmailId`, `MailboxId`, `ThreadId`,
  `EmailSubmissionId`, `CreateId`, `BlobId`).
- `SpfStatus`/`DkimStatus`/`DmarcStatus` accessors sit alongside the
  stable `String` wire form of `AuthVerdict`.
- DKIM `t=`/`x=` timestamps surface; `expired_at(now)` helper available.
- `verify()` is wall-clock-bounded at 10 s; timeouts become a synthetic
  `temperror` verdict, not a hang.
- **Blob dedup is race-safe** (`write_atomically` with per-call unique
  tmp + `ALREADY_EXISTS` reconciliation; concurrent writer test
  passes; `durability` flush is post-rename).
- **`QueryFilter` rejects unknown keys** (typo guard) at parse time.
- **`SmtpConfig` exposes `max_errors` and `read_timeout_secs`**;
  the session wall-clock lifetime caps at 5 min.
- **`Email/get` and `Mailbox/get` honor the `properties` subset**
  (RFC 8621 §6.1): unknown properties are `invalidArguments`,
  empty list = no properties, missing/None = all.
- **`ResponseTooLong` closes the connection** — persistently
  excessive line lengths are a client-misbehavior or DoS signal,
  not something to increment-and-wait on.
- **`AUTH` command returns `502 5.5.1`** per RFC 4954 (not the old
  `503 5.5.1`, which is "bad sequence", not "not implemented").
- MTA-STS / DANE / PGP key lifecycle / `*\/set` `destroy` handling /
  per-IP connection rate-limit / state timeout are explicitly **not**
  in this wave. Each has a tracking follow-up below.
