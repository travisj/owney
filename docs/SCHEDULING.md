# Scheduling Pages ("Book a meeting with me")

**Status**: Implemented and verified end-to-end 2026-07-15 (see What works below)

## What it is

A Calendly-style public booking flow built into the mailserver. An account
owner creates a page with availability rules; visitors hit
`https://server/schedule/{slug}` — no account, no auth — pick a slot, and
book it. Booking instantly creates a calendar event and emails both parties
a confirmation with a `text/calendar` (METHOD:REQUEST) invite: the visitor
via the real outbound pipeline (PGP/DKIM/queue), the owner via direct inbox
ingest (which also fires JMAP push).

## Data model (migration 20→21)

`scheduling_pages`: slug (server-wide UNIQUE, `[a-z0-9-]`), title,
description, target `calendar_id`, IANA `timezone`, `availability` (JSON,
below), `durations_mins` (JSON array), buffers, `min_notice_mins`,
`max_per_day`, `valid_from`/`valid_until` dates, `status`
(`active`/`paused`).

`bookings`: page/account/event references, visitor name/email/note,
start/end, `status` (`confirmed` now; `pending` reserved for a future
approval flow), `cancel_token` (reserved for future cancellation links).

### Availability JSON (version 1)

```json
{
  "version": 1,
  "weekly": {
    "mon": [{"start": "09:00", "end": "17:00"}],
    "fri": [{"start": "09:00", "end": "12:00"}, {"start": "13:00", "end": "17:00"}]
  },
  "overrides": {
    "2026-07-20": [],
    "2026-07-25": [{"start": "10:00", "end": "12:00"}]
  }
}
```

- Times are wall-clock in the page's timezone; absent weekday = unavailable.
- Default is Mon–Fri 09:00–17:00 — **weekends are excluded by default but
  fully supported**: add `"sat"`/`"sun"` windows.
- `overrides` replace a date's windows entirely; `[]` blocks the date.
- Unknown fields are rejected (`deny_unknown_fields`); windows must be
  sorted and non-overlapping.

### DST rules (chrono-tz; tested against America/Denver)

- Slots are generated by wall-clock stepping, so a fall-back day offers each
  wall time once — an ambiguous local time resolves to the **earliest**
  instant.
- A spring-forward gap time resolves to the first valid instant after the
  gap (deduped against the neighboring slot).
- Meetings are always their real duration in elapsed time.

## Public HTTP surface (unauthenticated by design)

- `GET /schedule/{slug}` — server-rendered booking page (vanilla HTML/JS,
  no build step). Owner-controlled strings are HTML-escaped.
- `GET /schedule/{slug}/slots?from&to&duration` — JSON:
  `{"timezone", "durationMins", "slots": [{"start", "end", "startLocal"}]}`.
  Range ≤ 31 days, horizon ≤ 60 days.
- `POST /schedule/{slug}/book` `{"start", "durationMins", "name", "email",
  "note"?}` → `201 {"bookingId", ...}`.

Semantics worth knowing:

- Unknown **and paused** slugs return a uniform 404 (no existence oracle).
- The server never trusts the picked time: `book` re-derives the slot grid
  (windows, notice, valid range) and rejects off-grid picks with 400.
- Conflicts are decided inside one storage transaction on the SQLite writer
  thread (`Storage::book_slot`): busy calendar events (across ALL the
  owner's calendars, buffer-widened), confirmed bookings, day quota, and a
  pause racing the booking. Losers get **409** — the race test
  (`booking_race_yields_exactly_one_winner`) proves exactly one concurrent
  winner.
- Rate limits (in-memory, per IP): book 10/hour, slots 60/min → 429. Client
  IPs come from `ConnectInfo` (serve now uses
  `into_make_service_with_connect_info`).

## Owner surfaces

JMAP, under capability `urn:owney:params:jmap:scheduling`:

- `SchedulingPage/get` / `SchedulingPage/set` (create/update; pause with
  `{"status": "paused"}`; destroy unsupported — pause instead). Create
  defaults: slug from the address localpart, title "Meet with {name}",
  UTC, Mon–Fri 9–5, one 30-minute duration.
- `SchedulingBooking/get` — list bookings (optionally by `pageId`).

CLI (also used by the lab):

```bash
owneyd admin create-scheduling-page alice@example.com <calendar_id> \
    --timezone America/Denver [--slug s] [--title t] [--duration 30]
owneyd admin scheduling-pages alice@example.com
owneyd admin bookings alice@example.com
```

## Trying it in the lab

```bash
scripts/lab.sh up
scripts/lab.sh admin alice create-calendar alice@alice.local Meetings   # note id
scripts/lab.sh admin alice create-scheduling-page alice@alice.local <cal_id> \
    --timezone America/Denver
open http://127.0.0.1:8381/schedule/alice        # or curl
curl -s 'http://127.0.0.1:8381/schedule/alice/slots?from=2026-07-20&to=2026-07-20'
curl -s -X POST http://127.0.0.1:8381/schedule/alice/book \
  -H 'content-type: application/json' \
  -d '{"start":<slot>,"durationMins":30,"name":"Bob","email":"bob@bob.local"}'
scripts/lab.sh inbox alice          # owner notification (direct ingest)
scripts/lab.sh inbox bob            # visitor confirmation (real SMTP)
scripts/lab.sh admin alice bookings alice@alice.local
```

## What works (verified live 2026-07-15, two-instance lab)

- Public page + slots without auth; correct Denver-timezone slot instants.
- Book → 201; identical re-book → 409; booked slot vanishes from `/slots`.
- Owner notification landed in alice's inbox (direct ingest); visitor
  confirmation traversed real SMTP to bob with the `invite.ics` part intact —
  and bob's enrichment worker detected it as a `calendarInvite` server
  attribute (the two features compose).
- A `create-event` busy block removed the overlapping slots from `/slots`.
- Proving tests: `cargo test -p owney-storage scheduling` (race, authz,
  quota), `cargo test -p owney-api --lib schedule` (DST boundaries, ICS
  folding, MIME round-trip, rate limiter), `cargo test -p owney-api --test
  scheduling_http` (wire-level flow + boundaries), `cargo test -p
  owney-jmap-mail --lib scheduling` (cross-account negatives).

## Known limitations

- **Recurring owner events don't block slots** beyond their base occurrence:
  no rrule engine exists anywhere in the workspace (pre-existing; the
  `rrule` column round-trips unexpanded).
- No cancellation flow (`cancel_token` stored but unused) and no approval
  flow (`pending` status reserved).
- Rate limiter is per-process memory; resets on restart. No
  X-Forwarded-For handling — direct-serve or loopback assumed.
- Email failures after a successful booking are logged, not retried by the
  booking layer (the visitor path is queue-durable once submitted; the
  owner-side ingest is one-shot). The calendar event is the source of truth.
- No `Email`-style change tracking for pages/bookings (no modseq); owner
  clients poll `SchedulingBooking/get` or watch the inbox notification.
- Pre-existing upstream gap: `delete_account` does not cascade
  `calendars`/`calendar_events` (FK violation risk on account deletion);
  the new `bookings`/`scheduling_pages` tables ARE in the cascade.
